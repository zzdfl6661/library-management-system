# 图书馆管理系统重构 - 需求分析计划

> **文档版本**: v1.0
> **生成日期**: 2026-06-10
> **基准数据库**: `/workspace/library.sql`（旧版设计）

---

## 一、核心问题概述

用户要求**以 `library.sql`（旧版数据库架构）为准**重构系统，同时对手写原型描述的功能进行调整。当前代码库的数据库架构（`schema.sql` + 实体类）与 `library.sql` 存在重大差异，需要进行深度改造。

## 二、数据库架构冲突详解

### 2.1 表命名与存在性差异

| library.sql 表名 | 当前代码库表名 | 冲突说明 | 解决方案 |
|:---|:---|:---|:---|
| `admin` | `sys_user` | 表名完全不同，字段结构不同 | 改用 `admin` 表结构，保留管理员三种类型（证件管理/采编/流通） |
| `student` | `student` | 表名相同但字段差异大 | **以 library.sql 为准**：恢复 sno/username/type/collage/major/birth/originPlace/password 字段，移除 max_borrow_count |
| `libcard` | `library_card` | 表名不同，字段差异大 | **以 library.sql 为准**：恢复 cardNo/sno/sname/type/collage/major/birth/originPlace/cardStatus/times 字段，用 cardNo 做主键 |
| `book` | `book` | 表名相同但字段不同 | **以 library.sql 为准**：恢复 ISBN/bname/author/publisher/introduction/pubDate/clcNum/bookStatus 字段，用 id 或 ISBN 做关联 |
| `bookcopy` | `book_copy` | 表名不同，字段不同 | **以 library.sql 为准**：恢复 barCode/ISBN/place/status/oldStatus 字段，用 barCode 做主键 |
| `borrowrec` | `borrow_record` + `fine_record` | 当前拆分了两张表，旧版是一张表含罚款字段 | **以 library.sql 为准**：合并为单表 `borrowrec`，包含 serNum/sno/barCode/borDate/retDate/realRetDate/retStatus/ovdDays/fineMoney/fineStatus/paySerNum |
| `payrec` | `payment_record` | 表名不同，字段不同 | **以 library.sql 为准**：恢复 serNum/sno/payAmount/payDate 字段 |
| `cardrec` | **无此表** | 当前完全缺失借书证操作记录表 | **新增** `cardrec` 表：serNum/sno/originCardNo/newCardNo/opType |
| `notice` | **无此表** | 当前缺失公告表 | 暂不实现公告功能（需求中未提及） |

### 2.2 关联方式差异

| 关联关系 | library.sql 方式 | 当前代码库方式 | 解决方案 |
|:---|:---|:---|:---|
| 借书证 ↔ 学生 | `libcard.sno = student.sno`（字符串学号） | `library_card.student_id = student.id`（id 外键） | 改用 `sno` 字符串关联 |
| 图书副本 ↔ 图书 | `bookcopy.ISBN = book.ISBN`（ISBN 字符串） | `book_copy.book_id = book.id`（id 外键） | 改用 `ISBN` 字符串关联 |
| 借阅记录 ↔ 学生 | `borrowrec.sno = student.sno` | `borrow_record.card_id → library_card.student_id` | 改用 `sno` 直接关联 |
| 借阅记录 ↔ 副本 | `borrowrec.barCode = bookcopy.barCode` | `borrow_record.copy_id = book_copy.id` | 改用 `barCode` 直接关联 |
| 罚款 ↔ 缴费 | `borrowrec.paySerNum`（可选关联） | `fine_record.student_id ↔ payment_record.student_id` | 改用 borrowrec 内部字段记录罚款，payrec 表做缴费记录 |

### 2.3 状态字段设计差异

| 状态字段 | library.sql | 当前代码库 | 解决方案 |
|:---|:---|:---|:---|
| 图书副本状态 | `status INT` (0借出, 1可借, 2注销) | `status VARCHAR` (AVAILABLE/BORROWED/DAMAGED/WITHDRAWN) | 改用 INT 状态，对应 0借出/1可借/2注销 |
| 借书证状态 | `cardStatus VARCHAR` (正常/挂失/注销等) | `status VARCHAR` (NORMAL/LOST/CANCELLED) | 保持中文状态值 |
| 图书状态 | `bookStatus INT` (0下架, 1上架) | 无此字段 | 恢复 bookStatus 字段 |
| 还书状态 | `retStatus ENUM` ('按时还','超时还','未归还') | `is_overdue TINYINT` | 改用 ENUM 中文值 |
| 罚款状态 | `fineStatus ENUM` ('已缴','未缴') | `is_paid TINYINT` | 改用 ENUM 中文值 |

## 三、功能需求与现有实现的差距

### 3.1 证件办理模块（需求 vs 现状）

| 功能点 | 需求描述（手写原型） | 当前实现 | 差距 |
|:---|:---|:---|:---|
| **新办证** | 学号查询→获取学生信息→输入借书证号→确认办理；支持批量 | 输入学号→自动生成 cardNo→创建 | 需改为**手动输入借书证号**，增加批量办理功能，展示姓名/学号/籍贯/是否已办 |
| **挂失** | 学号查询→显示学生基本信息（姓名/性别/专业/班级/身份证号/证件状态）→确认挂失 | 按卡号挂失，无学生信息展示 | 需改为按学号查询，展示完整学生信息，修改状态同时生成 cardrec 记录 |
| **补办** | 学号查询→学生基本信息→输入新借书证号→注销旧卡+生成新卡 | 必须先挂失才能补办，自动生成卡号 | 需改为**手动输入新卡号**，可直接补办（不一定需要先挂失），同步生成 cardrec |
| **注销** | 学号查询→注销状态；支持批量 | 按卡号注销，无批量 | 需增加批量注销功能，同步生成 cardrec 记录 |
| **扫码输入** | 所有卡片/证件号码输入框支持外接设备扫码 | 无扫码支持 | 使用 `input` 自动聚焦，扫码枪模拟键盘输入即可 |
| **操作记录表** | 每次操作生成 cardrec 记录 | 无 cardrec 表 | 新增 cardrec 表并在每次操作时写入 |

### 3.2 检索与读者中心（需求 vs 现状）

| 功能点 | 需求描述 | 当前实现 | 差距 |
|:---|:---|:---|:---|
| **首页推荐三区** | 热读推荐 + 借书明星 + 新书速递 | 首页有学生排行榜 + 图书排行榜，但字段不对 | 需要调整为三个独立板块：热读（借阅量最高的书）、借书明星（借阅最多的读者）、新书（最新入库的书） |
| **登录按钮** | 首页右上角有登录入口 | 有登录页面但首页入口不明显 | 明确登录入口 |
| **个人信息管理** | 查看/编辑个人基础信息 + 修改密码（libcard 需加 password 字段） | 无独立个人信息页，密码在 student 表 | 需要新增个人信息页面，按需求在 libcard 表增加 password 字段 |
| **借阅记录查询** | 按借书日期倒序；表格列：序号/书名/作者/借书日期/应还日期/实际还书日期/还书状态/罚款金额/罚款状态；分页 + 未缴罚款总额 | 有我的借阅页面，字段不完整 | 需要补充：作者、实际还书日期、还书状态、罚款金额；增加分页；底部显示未缴罚款总额 |
| **罚款缴纳** | 默认显示未缴罚款；表格列：序号/书名/作者/借书日期/应还日期/实还日期/罚款金额/状态/复选框；勾选后自动计算合计金额；支付按钮；支持导出 | 有我的罚款页面，字段不完整 | 需补充：书名、作者、借书日期、应还日期、实还日期；增加多选复选框；增加自动合计；增加导出功能 |

### 3.3 流通部模块（需求 vs 现状）

| 功能点 | 需求描述 | 当前实现 | 差距 |
|:---|:---|:---|:---|
| **借书流程** | 1) 输入借书证号→检查（弹框显示原因或可借书数量）；2) 读者信息+条形码输入+确认；3) 扫码自动录入并解析书籍信息；4) 达到上限时禁止继续；5) 确认后生成借阅记录，更新副本状态为"借出" | 有借书页面，逻辑基本一致 | 需要优化：弹框确认、扫码后自动解析并展示书籍信息（条码号/ISBN/书名/作者/出版社/藏书位置）、达到上限时弹出提示 |
| **还书流程** | 输入图书条码→确认→自动计算超期天数和罚款金额→更新借阅记录和副本状态→提示"还书成功" | 有还书页面，逻辑基本一致 | 需要明确显示超期天数和罚款金额计算结果 |
| **无需登录** | 流通部操作不需要登录 | 当前需要 CIRCULATION 角色登录 | 改为可公开访问（或简化认证） |

### 3.4 采编部模块（需求 vs 现状）

| 功能点 | 需求描述 | 当前实现 | 差距 |
|:---|:---|:---|:---|
| **图书信息管理** | 表格列：ISBN/书名/作者/出版社/出版日期/编辑/副本数；操作：下架/编辑/删除；分页；已下架图书下架按钮变灰 | 仅实现新增图书入库 | 需完整实现：图书列表展示、编辑、下架、删除、分页、状态控制（已下架不可再下架） |
| **图书副本管理** | 入口：点击某本书的"副本"按钮；顶部显示图书基本信息 + 新增副本按钮；表格列：条码号/ISBN/藏书位置/状态/编辑/注销 | 无此页面 | 需完整实现：独立副本管理页面、新增副本（录入条码号+藏书位置）、编辑、注销 |
| **旧书注销** | 修改 bookcopy.status = 2（注销），不再参与借阅 | 当前有报废功能（DAMAGED） | 对齐状态值（改为 2），并确保检索时不显示注销副本 |
| **图书下架** | 修改 book.bookStatus = 0（下架），该书所有副本不可借阅，检索完全隐藏；在编册入库界面可见 | 当前有下架功能，批量更新副本状态为 WITHDRAWN | 需改为更新 book 表的 bookStatus=0，检索时过滤 bookStatus=0 的图书；采编界面仍可查看 |

## 四、需要向用户确认的问题

> ⚠️ **以下问题需要用户确认后才能继续开发**

### 问题1：学生详细信息的来源
**描述**: `library.sql` 中 `student` 表有 `type/collage/major/birth/originPlace` 字段，`libcard` 表也冗余存储了这些字段。需求中挂失功能需要展示"姓名/性别/专业/班级/身份证号"。

**疑问**: 
- `student` 表没有"性别""班级""身份证号"字段，但需求的挂失界面需要展示这些信息。是否需要在 `student` 表中新增这些字段？
- libcard 表的冗余字段是否仍需要（用于性能优化），还是只保留在 student 表中即可？

**推荐方案**: 在 `student` 表中新增 `gender`(性别)、`className`(班级)、`idCard`(身份证号) 字段；`libcard` 表保留冗余字段用于快速展示。

### 问题2：借阅上限的实现方式
**描述**: `library.sql` 中 `libcard.times` 字段记录已借数量，但没有"最大借书数"字段。

**疑问**: 是否根据 `student.type` 字段值（如"本科生/研究生/教师"）来动态判断最大借书数？还是在 `libcard` 或 `student` 表增加 `maxBorrow` 字段？

**推荐方案**: 根据 `student.type`（本科生5本/研究生10本/教师15本）动态判断，无需新增字段。

### 问题3：图书表的主键选择
**描述**: `library.sql` 中 `book` 表用 `id` 自增做主键，但 `bookcopy` 表通过 `ISBN` 字段关联图书。

**疑问**: 是否可以假设每本书的 ISBN 唯一？如果同一 ISBN 有多条 book 记录会有问题。

**推荐方案**: 以 `ISBN` 作为图书表的业务唯一键，book.id 继续保留做数据库自增主键。

### 问题4：流通部"无需登录"的安全性
**描述**: 需求明确说"流通部模块（借书/还书，无需登录）"。

**疑问**: 这是否意味着借书/还书操作完全公开？还是用其他方式（如工号/密码）简单认证？

**推荐方案**: 借书/还书操作不需要管理员登录，但建议记录操作员信息（可选项）。

### 问题5：扫码枪输入支持
**描述**: 需求提到"支持外接设备扫码输入所有卡片/证件号码"。

**疑问**: 扫码枪通常模拟键盘输入，只要输入框有焦点即可自动输入。是否需要做额外处理（如自动回车提交、长度校验等）？

**推荐方案**: 输入框自动聚焦，扫码后自动触发回车事件提交查询/确认。
